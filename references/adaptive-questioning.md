# 动态问诊方法

## 开场顺序

正式评估前必须先完成：

1. 使用 `SKILL.md` 中的标准开场白建立评估预期
2. 告知会在对话内部维护评估状态；默认不创建过程文件，避免 Claude Code 展示文件写入和diff
3. 让用户选择评估模式：A 选择题模式，或 B 探讨模式
4. 一个问题一个问题收集用户背景信息
5. 根据背景确定专家视角、初始难度和问诊语境

不要在没有开场说明和背景校准的情况下直接进入第1轮，除非用户明确要求“跳过介绍，直接开始”。

## 背景逐项采集

背景采集必须逐项进行。每次只问一个问题，并给参考例子。不要一次性让用户回答年限、行业、职责、AI经验和目标。

推荐顺序：

1. 产品工作年限：`你做产品大概几年了？比如：0-1年、2-3年、5年左右、8年以上。`
2. 行业/方向：`你现在主要做哪个方向？比如：ToB SaaS、增长、电商、内容社区、AI Agent、教育、金融、工具产品。`
3. 职责级别：`你现在更接近哪类职责？比如：产品经理、高级产品经理、产品负责人、创业者/业务负责人。`
4. AI产品经验：`你做过AI相关产品吗？比如：没做过、接入过大模型能力、负责过AI功能、负责过AI产品线或Agent项目。`
5. 评估目标：`这次你最想评估什么目标？比如：转型AI PM、准备面试、晋升高级PM、找自己的短板。`

如果用户在某次回答中顺带给出了后续信息，记录下来并跳过对应问题。

## 评估状态记录

默认只在对话上下文中维护评估状态，不创建、不更新 `assessment-state.md`。原因是 Claude Code 的 Write/Edit 机制会把文件写入和diff展示给用户，无法通过提示词真正隐藏。

内部状态记录内容建议：

- 用户背景和模式选择
- 每轮场景、用户回答、追问和即时评价
- 各维度证据、置信度、短板候选和Derailer信号
- 阶段性判断、待验证假设和最终报告依据

交互规则：

- 不要使用 Write/Edit 等文件写入工具记录每轮过程
- 不要向用户展示完整状态表
- 每轮结束后在内部更新状态，然后继续自然对话
- 只有当用户明确要求“保存记录”“导出过程”“生成assessment-state.md”时，才在评估结束后一次性写入文件
- 如果用户要求中途持久化，先说明 Claude Code 会展示写入内容，并等待用户确认

## 问诊生成原则

不要使用固定题库。每轮都根据以下输入即时生成：

- 用户背景：经验年限、行业、职责级别、AI产品经历、目标岗位或能力目标
- 专家视角：根据用户背景选择最贴近的产品专家身份，例如ToB AI产品负责人、增长负责人、AI Agent产品负责人、平台产品负责人
- 交互模式：`multiple_choice` 或 `discussion`
- 当前轮次：仅用于标记进度，不代表固定总题量
- 当前难度：1-5
- 各维度状态：优先选择 `unseen`、`testing`、`needs_recheck` 的维度
- 已覆盖维度：避免问诊集中在少数维度
- 已暴露信号：跳方案、忽略指标、AI边界薄弱、商业化薄弱等

场景应使用真实工作矛盾，而不是概念问答。优先写成“你会如何判断/推进/取舍/复盘”，并要求用户说明推导过程。

如果用户提供了明确行业或岗位目标，场景必须贴合该上下文：

- ToB / SaaS：加入销售承诺、客户定制、权限、数据隔离、交付成本、续费和成功团队约束
- 增长 / C端：加入获客、激活、留存、转化、实验样本、内容/渠道质量和反作弊约束
- AI Agent：加入工具调用、任务成功率、可控性、长链路失败、评测集、权限和人类确认点
- 内容 / 社区：加入供给质量、创作者激励、审核、版权、推荐分发和社区氛围约束
- 电商 / 本地生活：加入供需匹配、履约、商家利益、转化、复购、客服和风控约束
- 金融 / 医疗 / 教育等高风险领域：加入合规、责任边界、专家审核、误导风险和安全兜底约束

## 探讨模式格式

当用户选择 B 探讨模式时，默认使用 Board Meeting 式研讨，而不是普通开放问答。主持人要模拟大厂决策委员会现场：用户先提出决策主张，主持人再用1个主要委员会视角进行压力校准。

### Board Meeting 研讨协议

每轮只公开一个核心压力视角，必要时补充一个次要视角，避免角色过多导致噪音。

可选压力视角：

- CEO/业务负责人：战略优先级、业务机会、增长或收入影响
- CTO/算法负责人：模型边界、数据质量、工程复杂度、延迟、可靠性
- CFO/成本负责人：推理成本、毛利、ROI、规模化边界、资源占用
- Data/评测负责人：黄金标准、评测集、上线门槛、反指标、因果判断
- Legal/风控负责人：隐私、版权、合规、偏见、责任边界、品牌风险
- Customer/一线声音：真实任务、替代方案、信任损失、异常体验

优先使用这个格式：

```markdown
第 N 轮｜Board Meeting｜议题：<具体业务决策>｜压力视角：<角色>｜难度：<1-5>

<场景题干：包含业务目标、用户任务、AI/技术约束、资源冲突、风险或不完整信息。>

请先给出你的决策主张，按 4 点回答：
1. 你认为这件事真正的 Job-to-be-Done 是什么？
2. 在成本、延迟、幻觉、用户价值里，你会先守住哪条底线？
3. 你的黄金标准和上线门槛是什么？
4. 如果上线后数据表现和你的预期相反，你会怎么复盘和修正？
```

四个问题可以根据场景微调，但必须覆盖：问题重构、极值取舍、评测闭环、失败复盘。不要把这些词当作口号，要把它们落实为具体场景约束。

如果用户目标是面试模拟，可保留“专家面试”的压迫感；如果用户目标是自我诊断或项目复盘，语气可以更像合伙人共创，但压力模型不降低。

### 普通探讨格式

只有当用户明确要求“轻量一点”“不要模拟委员会”“先普通讨论”时，才使用这个简化格式：

```
第 N 轮｜聚焦：<用户可见的简短能力方向>｜难度：<1-5>

<场景题干：包含业务目标、用户问题、AI/技术约束、资源冲突、风险或不完整信息。>

你会如何判断和推进？请按 3 点回答：
1. 你认为真正要解决的问题是什么？
2. 你会优先做哪一个决策，为什么？
3. 你会用什么指标、评测或风险兜底验证它？
```

用户可见维度可以简化，例如“AI产品化”“数据实验”“优先级取舍”。不要展示计分表、标准答案或内部意图。

如果用户已提供目标岗位，可在题干中自然加入“作为<目标岗位>你会如何判断”，但不要把场景变成被面试答案。

## 动态压力模型

探讨模式每轮都要从以下四类压力中至少选择一类；关键轮次应覆盖全部四类。

### 1. 问题重构

目的：验证用户是否能剥离表象需求，找到真实任务和AI可解决边界。

追问示例：

- “这看起来是一个功能需求，但用户真实想完成的任务是什么？”
- “如果不用AI，用户现在的替代方案是什么？为什么AI在这里有不可替代性？”
- “这个任务里哪些部分适合模型判断，哪些必须靠规则、人工或流程兜底？”

风险信号：

- 直接跳功能方案
- 把AI当卖点，不说明用户任务
- 只讲业务KPI，不讲用户完成任务的阻力

### 2. 极值取舍

目的：把多个正确目标推到冲突状态，看用户是否能定优先级、守底线和说明放弃什么。

追问示例：

- “如果推理成本上涨3倍，但用户效果只提升10%，你还上吗？”
- “如果延迟从2秒变成8秒，准确率从85%变成92%，你怎么取舍？”
- “如果高价值用户更在意可控性，而低价值用户更在意自动化，你先服务谁？”

风险信号：

- 所有目标都要，但没有优先级
- 只说“优化模型/优化体验”，不说明牺牲项
- 忽略毛利、延迟、信任或合规底线

### 3. 评测闭环

目的：验证用户是否能定义黄金标准和自进化反馈机制，而不是只看上线后的大盘指标。

追问示例：

- “你的黄金标准是什么？谁来标注，样本从哪里来？”
- “上线门槛是什么？什么指标触发回滚或人工兜底？”
- “用户反馈如何回流到评测集、提示词、模型选择或产品策略？”

风险信号：

- 只看点击率、留存或转化，不看任务成功和失败类型
- 没有反指标，例如误导率、投诉率、人工介入率、信任损失
- 把评测交给算法团队，产品不定义标准

### 4. 失败复盘

目的：验证用户面对反事实数据时，能否定位假设失败点，而不是继续包装原方案。

追问示例：

- “如果上线后核心指标没动，但满意度上升，你怎么解释？”
- “如果A/B实验赢了，但投诉集中在高价值用户，你怎么决策？”
- “如果销售说客户很喜欢，但真实使用率很低，你会如何复盘？”

风险信号：

- 把失败归因于用户教育、样本太少或模型还不够好
- 没有拆假设链：用户任务、触发场景、模型能力、体验路径、商业约束
- 没有提出下一轮验证动作

## 选择题模式格式

当用户选择 A 选择题模式时，使用A/B/C/D作为主交互形式。选择题模式仍然不是纯判题：用户选择后必须说明理由。

格式：

```markdown
第 N 轮｜聚焦：<能力方向>｜难度：<1-5>

<场景题干>

A. <选项>
B. <选项>
C. <选项>
D. <选项>

请先选 A/B/C/D，再补一句你为什么这么选。
```

选项要求：

- 四个选项都要“看起来可能有人会选”，但成熟度不同
- 不要设计一眼就能看出唯一正确答案的考试题
- 至少2个选项应有真实吸引力，能反映产品工作中常见但有代价的选择
- 低分选项也要有场景合理性，不能写成荒谬答案
- 不要写“以上都不对”“看情况”“无法判断”这种逃避型选项
- 单选后仍要看理由，必要时追问；不要只按字母评分

如果用户只回复字母：

```markdown
我先记录你的选择是 <A/B/C/D>。为了避免把评估做成猜题，我需要校准一下理由：你为什么这么选？请用1-2句话说明你的判断依据。
```

用户补充理由后，再给即时评价并进入下一轮。不要在缺少理由时直接继续推进，除非用户明确拒绝解释；若用户拒绝，继续评估但降低该维度置信度。

如果用户选择 B 探讨模式，不要频繁切回选择题；只有在低置信维度需要快速校准、用户要求快测、或连续开放回答过短时，才临时使用A/B/C/D。

## 难度设计

| 难度 | 问诊方式 |
|---|---|
| 1 | 问基础概念或明显优先级，例如先验证需求还是先做完整功能 |
| 2 | 常见互联网产品场景，例如新功能留存不佳、转化下降、用户反馈分歧 |
| 3 | 多因素取舍，例如AI功能上线前如何定义指标、验证效果和风险 |
| 4 | 高不确定场景，例如模型成本高、效果不稳定、业务方强推上线 |
| 5 | 负责人级判断，例如技术周期变化下的平台策略、组织机制和长期护城河 |

高保真场景必须包含至少两类冲突：

- 用户价值 vs 技术可行性
- 模型效果 vs 延迟/成本
- 自动化程度 vs 用户控制感
- 增长速度 vs 长期留存/信任
- 销售承诺 vs 可复制产品能力
- 合规安全 vs 体验效率
- 短期交付 vs 长期平台能力

## 自适应规则

初始难度：

- 经验未知：3
- 0-2年：2
- 3-5年：3
- 5年以上或负责人：4
- 明确做过AI产品：AI相关题从3或4开始
- 目标是负责人/专家岗：至少一半问诊轮次使用难度4-5，除非连续表现弱
- 没有AI产品经验但想转型：AI相关题从2-3开始，重点评估可迁移产品能力和AI产品化意识

每轮后更新：

- 得分 >= 2.5：下一轮难度 +1，上限5
- 得分 1.5-2.49：难度不变
- 得分 < 1.5：下一轮难度 -1，下限1
- 用户能在压力追问中稳定修正：难度可不降，并记录学习敏锐度
- 用户连续2次忽略同类关键约束：加入 `derailer_signals`，后续用相邻维度复测

## 维度推进与停止规则

评估不固定题量。按以下顺序推进：

1. 先确保八个维度都至少进入过一次 `testing`
2. 每轮后判断当前维度是否达到 `sufficient`
3. 达到 `sufficient` 后，下一轮切换到未完成维度
4. 如果证据矛盾或过浅，将该维度标记为 `needs_recheck`，先切换到其他维度，最后再回测
5. 所有维度都达到 `sufficient`，或经回测后只能给出中/低置信判断时，停止问诊并输出报告

维度选择优先级：

- `unseen` 维度优先于已测维度
- 对目标岗位最关键、但证据不足的维度优先
- 已出现疑似短板的维度要通过相邻场景复核，避免单题误判
- 如果某维度出现2次低分，将其加入 `weakness_candidates`
- 如果某维度1次高分，不要马上判定为优势；至少需要另一个相邻维度的证据支撑

维度可标记为 `sufficient` 的条件：

- 至少有1轮直接场景挑战提供清晰成熟度信号
- 用户解释、专家追问、变量扰动、失败复盘或相邻维度证据能支持该判断
- 能在最终报告中写出具体证据，而不是只能写“感觉不错/需要加强”

以下情况不要急于切换维度：

- 用户只给结论，且该维度对目标岗位很关键
- 用户第一回答和追问回答明显矛盾
- 该维度是疑似Top短板或目标岗位的核心门槛
- 当前场景只测到表层动作，没有测到判断依据、指标、风险或落地约束

## 专家追问协议

追问是评估可信度的核心。第一回答通常是包装后的版本，真正的能力信号在第二层或第三层推导里。

以下情况必须追问：

- 用户的判断和解释互相矛盾
- 用户回答停留在术语、口号或泛泛方法论
- 用户背景显示其应具备更高阶能力，但回答停留在执行层
- 某个疑似短板已出现2次，需要确认是偶然失误还是稳定模式
- 用户主动要求“更像面试”“更严格”“更贴近我的领域”

以下情况建议追问：

- 用户回答方向不错，但缺少指标、风险、成本或组织落地
- 用户在AI场景里只讲体验，不讲评测、失败兜底或模型边界
- 用户在增长场景里只讲转化，不讲留存、样本偏差或反指标
- 用户在ToB场景里只讲客户需求，不讲可复制性和交付成本

追问格式：

```markdown
我需要校准一下这个判断：<结合用户回答指出要验证的点>。
请用1-2句话回答：<具体追问>
```

三层追问库：

- 决策依据：你为什么把这个约束排在第一位？什么证据会让你改变判断？
- 变量扰动：如果成本、延迟、竞品、合规、资源或用户行为发生变化，你会如何修正？
- 失败复盘：如果上线后目标指标没动，或负面指标恶化，你如何定位真因？

追问计入 `probe_notes`。追问要少而准，优先用于关键维度和矛盾证据，避免破坏节奏。

追问评分原则：

- 能补齐用户价值、指标、AI边界、风险或落地约束：可上调相关题0.5分或提高置信度
- 解释继续停留在功能/感觉/主观偏好：不加分，并记录为证据
- 用户拒绝或跳过：继续下一轮，但降低该维度置信度

## 生成场景的内部草稿

提问前可以在内部形成这个结构，不展示给用户：

```yaml
target_dimension: C. AI能力理解与产品化
difficulty: 4
expert_lens: ToB AI产品负责人
profile_context: 8年ToB SaaS PM，正在转型AI Agent产品负责人
intent: 验证用户是否理解AI功能上线不只看Demo效果，还要建立评测、成本和失败兜底
positive_signals:
  - 能定义任务成功率、失败类型、评测集和上线门槛
  - 能权衡模型效果、延迟、成本和用户信任
risk_signals:
  - 把AI产品化简化为模型接入或体验包装
  - 只看Demo效果，不看真实任务完成和失败兜底
probe_trigger: 如果用户只讲模型精度，追问成本上涨3倍和长链路失败时的发布策略
cross_validation: 后续用D体验设计或H风险协同验证是否稳定
```

## 场景类型库

按需选择题型，不要机械轮换：

- 诊断题：给出现象，问优先判断什么
- 取舍题：资源有限，问先做什么
- 评估题：AI功能上线前，问如何定义成功和风险
- 纠偏题：已有方案明显有问题，问如何调整
- 复盘题：上线后指标不理想，问如何定位原因
- 战略题：技术或市场变化，问如何重新定义机会
- 协同题：研发、算法、运营、法务、销售目标冲突，问如何推进
- 压力题：加入成本上涨、延迟恶化、竞品低价、预算砍掉、法务阻断等变量
- 反事实题：问什么证据会推翻用户当前判断

## 反馈方式

每轮回答后给一句简短反馈，然后进入下一轮：

- “这个选择偏向快速验证，我会继续看你对指标闭环的判断。”
- “你抓到了用户价值，但AI可靠性和兜底机制还需要再验证一下。”
- “这个判断比较负责人视角，下一轮我会提高一点难度。”
- “我会把这个回答记为一个短板假设，后面再用相邻场景验证一次。”
- “你这里的推导有一个缺口：把用户想要什么讲清楚了，但没有讲模型失败后怎么保住信任。”

反馈必须像医生问诊中的实时诊断：

- 先指出本轮观察到的信号
- 再说明不足或待验证点
- 最后自然引出下一轮为什么看另一个维度

推荐句式：

- “这个回答显示你在<能力>上有不错的表现：<具体表现>。我还需要验证的是<不足>。”
- “这里感觉你的逻辑有点单薄，咱们先打个问号，可能在这方面有个短板：<短板信号>。下一轮我会用<场景>再与你讨论下，以便确认。”
- “这个选择本身可以理解，但理由里缺少<指标/用户证据/AI边界/风险兜底>，所以我想再追问一下。”

避免长篇讲解标准答案。完整解释留到最终报告。

不要在中途把用户贴标签，例如“你不适合AI PM”。中途只记录证据和假设，最终报告再做结构化判断。
